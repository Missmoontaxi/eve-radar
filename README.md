# Eve Events Radar — hosted app

Internal, interactive home for the Eve Events Radar. Lives at **`radar.moontaxilab.com`**
(its own Vercel project, separate from the public marketing site). Renders the scored
event list as a slick, filterable destination for the Eve team — not a static file.

> **Internal only.** The URL is unlisted and the app is marked `noindex`. There is no login —
> don't post the link anywhere public.

## What it does

- **Renders from `public/events.json`** — the data feed produced by the `events-radar` skill.
- **Color-coded by city** (NYC · SF · Austin · Online · National) on every card and pill.
- **Filter** by city, tier, type, owner, and cost — multi-select, combine freely, with live counts.
- **Search** across name, host, tags, city. **Sort** by Team picks (★), score, date, or city.
- **List & Calendar views** — tiered editorial list, or a week-banded agenda timeline.
- **★ Star to rank** — anyone can star an event; the tally boosts it in the "Team picks" sort.
  Counts are **shared across the team** (Vercel KV) and **survive a data refresh**.
- **Export** — CSV of the current filtered set, and per-event `.ics` calendar files.

## Architecture

```
radar.moontaxilab.com
├─ public/index.html · styles.css · app.js   static SPA, no build step
├─ public/events.json                         data feed (regenerated by the skill)
└─ api/stars.js                               serverless fn → Vercel KV (shared star counts)
```

Plain static files + one serverless function. No framework, no build — so "manual publish"
is just: regenerate the data, push.

## One-time setup

1. **Create the Vercel project** and point the `radar.moontaxilab.com` subdomain at it
   (Vercel dashboard → Add Domain).
2. **Create a KV store** (Storage → Create → KV / Upstash Redis) and **connect it to this
   project**. Vercel injects `KV_REST_API_URL` and `KV_REST_API_TOKEN` automatically — that's
   all `api/stars.js` needs. (See `.env.example`.)
3. `npm install` (installs `@vercel/kv`).

If KV isn't configured, the app still runs — stars just fall back to each person's browser
(`localStorage`) instead of being shared.

## Refreshing the radar (the publish flow)

The radar is regenerated by the skill, then republished. From the repo root:

```bash
# 1. In Claude (in the eve-gtm repo): “run the events radar”
#    → writes a fresh events.json to that repo's outputs/ folder
# 2. From this app repo, import it and deploy:
npm run import -- <path-to-eve-gtm>/outputs/events.json   # validates + copies to public/events.json
vercel deploy --prod
```

> `<path-to-eve-gtm>` is wherever the `eve-gtm` repo is checked out. On Paula's machine
> that's `../Eve` relative to a sibling layout — adjust to your own path.

`npm run import` checks the file shape, confirms every event has a stable `id`, and prints a
tier summary before writing. Star counts are keyed by that `id`, so the team's picks carry
over to the new data as long as an event's name + date are unchanged.

> Per the site's standing rules, **ask Paula before deploying to production.** Branch/preview
> deploys (`vercel deploy` with no `--prod`) are fine for sharing work-in-progress.

## Local preview

```bash
cd Website/eve-radar
npm install
vercel dev          # serves the app + /api/stars locally (needs KV env for shared stars)
```

Or serve `public/` with any static server to preview the UI (stars fall back to localStorage).

## Data schema (`events.json`)

```jsonc
{
  "meta": { "run_date", "window_start", "window_end", "window_days",
            "source_types": [], "total", "counts": { "Top Events", "Worth it", "Monitor" } },
  "events": [{
    "id",            // stable slug+date — the KV key for star counts. Do not change its formula.
    "rank", "name", "url", "score", "breakdown",
    "tier",          // Top Events | Worth it | Monitor
    "city",          // NYC | SF | Austin | Online | National | other  (drives colour-coding)
    "city_raw", "venue", "host_org", "mode",
    "distribution",  // Direct | Partner
    "format", "owner", "cost", "outcome",
    "date_start", "date_end", "date_label", "topic_tags": [], "source", "description"
  }]
}
```

The `id` formula lives in `scripts/score_events.py` (`event_id()`). Changing it orphans the
stars stored against old ids in KV — leave it alone unless you intend to reset the tally.
